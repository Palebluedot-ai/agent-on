# 能力真相表:动工前先说清「到底有没有」

> 职责边界:本篇管「动重大实现前,先产出一张诚实的能力清单并让用户确认」;防的是 AI 假装某功能已实现就闷头往下建。
> 源流:二代 communication-governance-playbook/imported/evolving-memory-system/04_protocols/global-inscope-lock-protocol-v1.md §Truth Table Requirement,2026-07-07 批二移植

## 一、防什么:AI 最爱假装「已有功能」

要动任何重大实现之前,AI 必须先产出 / 更新一张**能力真相表**——把要用到的每项能力逐条标注状态。真相表**未经用户确认,不许开工**。

它防的是一类高频翻车:AI 在没核实的情况下,把「听起来应该有」当成「已经有」,于是在一个不存在的地基上往上盖。等盖到一半才发现某个接口根本没接、某个函数只是个空壳,已经返工一大片。真相表就是逼你**在动手前把「有 / 没有」摊到桌面上**,而不是嘴上说「这个应该没问题」(违反 L2:完成 = 贴命令实际输出)。

## 二、四种状态(每项能力必须选一个)

| 状态 | 含义 | 判定标准 |
|---|---|---|
| **已自动化** | 代码里已实现且能自动跑通 | 有对应测试 / 命令,能贴出实际输出 |
| **手动** | 靠人手工做,没自动化 | 步骤存在但要人操作,不能无人跑 |
| **未实现** | 压根还没有 | 没代码 / 只有空壳 / 只在设想里 |
| **依赖环境** | 逻辑写了,但要外部条件才生效 | 依赖 key / 外部服务 / cron / 部署环境 |

「依赖环境」单列很关键——它是最会骗人的一档:本地看着「有」,换个环境就塌(没 key、服务没连、cron 没挂)。铁律 L7「外部服务第一天」(见 BOOTSTRAP.md)防的就是这一类。

## 三、模板 + 三行示例

```md
## 能力真相表 —— <项目/模块名> @ <日期>

| 能力 | 状态 | 证据 / 缺口 |
|---|---|---|
| 用户登录鉴权 | 已自动化 | `pytest tests/auth` 12 passed(附输出) |
| 客户数据导出 | 手动 | 得人工跑脚本 + 手工核对,未接进流程 |
| 支付回调对账 | 未实现 | 只有函数签名,body 是 TODO |
| 短信通知 | 依赖环境 | 逻辑就绪,缺 TWILIO_KEY;prod 未配 |
```

规矩:**「证据 / 缺口」列不许空**。标「已自动化」就贴能跑的命令;标其余三档就写清缺什么、卡在哪。空着 = 没核实 = 不算数。

## 四、什么时候必须出这张表(三个时机)

1. **接外部服务前**——接数据库 / 支付 / 第三方 API / 云服务之前。真实载荷形状、key、部署环境全是「依赖环境」档的重灾区,先摊开。
2. **接手他人代码前**——继承别人的仓 / 模块时,别信 README 说「支持 X」。逐条核实哪些真能跑、哪些是画的饼。
3. **声称「已有功能」前**——凡是要说「这个我们已经有了,直接用」,先用真相表自证:有,就贴证据;没有,就老实标未实现。

一句话带走:**真相表不是文档任务,是一道诚实闸门——它把「我以为有」和「确实有」之间那道最贵的缝,在动工前就缝死。**
