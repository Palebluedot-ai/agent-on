# 案例 16:一代标本:5200 行引擎蓝图 0 行代码(含 non-drift 可执行基线被 revert)

> 层级:L2 教训 | 来源:一代仓 agent-orchestration-playbook + 二代 non-drift 仓 | 入册:2026-07-07 批三

## 症状
一代仓「自建编排层」写满 5200+ 行设计物(手写 md+schema+脚本实测 5251 行,其中纯文档 4317 行,不含自动生成报告),`implementation/phase0/` 下 core / lanes / roles / prompts / tests 五个目录全是空壳,引擎 0 行落地代码。同一条主线上,二代 non-drift 仓的 git 历史里也留了个微标本:`feat: add executable non-drift protocol baseline` 提交后紧接着一条 `Revert`——可执行实现被回退,纯文档协议活了下来。

## 根因
这不是没时间实现,是方向选错:两处都在造「引擎」——一个要自建运行时兜底弱模型,一个要把协议做成可执行守卫。可当工具能力(Claude Code / Codex)已经覆盖了编排与守卫,自建引擎就成了空转的重复投资,天然长不出代码。反直觉处:引擎会死,纪律会活——两处死的都是「要造的东西」,活的都是「要遵守的约束」(契约 schema、DoD、单一写者、三态转移、真相源分层)。

## 修法
把两仓当证据层原地归档,只提炼「活下来的纪律」进 agent-on,引擎蓝图全部进死亡名单;正资产(「漂移是核心敌人」的问题定义、真相源分层)单独收编——问题定义领先于工具能力,方向从来是对的,错的只是「用自建引擎去实现它」。

## 可复用规则
动手造编排层/守卫/运行时之前,先问「现有工具(Claude Code / Codex)是不是已经覆盖了这个能力」——覆盖了就立纪律(契约 + 门禁 + 单写者),别造引擎;能写成一条要遵守的约束的,就不要写成一个要维护的系统。

## 已固化到哪
部分固化:README「它从哪来(三代演化)」节已把一代教训写成演化史,`playbook/orchestration-future.md` 承接「何时才该上编排」的判断;死亡名单在 snapshot 融合地图 §6。缺口:non-drift revert 这个微标本、以及「问题定义可以领先工具能力」这条正资产,尚无独立的祈使规则落点。待固化:建议把上面「可复用规则」那句进 playbook/meta-principles.md 作为一条元原则(与原则一「自己开发自己」并列),让下次「想造引擎」时有硬拦点——中风险,留给首次消化会话拍板。
