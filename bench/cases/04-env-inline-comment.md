# 案例 04：.env 行内注释：三种解析器三种行为

> 层级：L2 教训 | 来源：Euan 实战 2026-07-03 | 入册：2026-07-07 批三

## 症状
同一请求，curl 和 vitest 全 200，唯独经 vercel dev 必 503；而用户的另一个消费方又正常——key、上游、网络全都排除了，典型「A 路通 B 路不通」的薛定谔故障。

## 根因
`.env` 里那行值带了行内注释（`model=x-ai/grok-4.3    # 与旧 PWA 管线一致…`）。反直觉处：三种解析器对同一行三种行为——vercel 用的 dotenv 解析器把 `# …` 连同前面的空格一起并进了值，而 shell `source` 和 node `--env-file` 都会把行内注释剥掉。所以只有走 dotenv 那条路的进程拿到了脏值。

## 修法
先二分定位（绕开进程直跑=好，嫌疑锁进程）→ 给进程加一行 env 自曝，当场抓到值尾巴挂着注释串。规范落地：`.env` 值行禁止行内注释，注释一律独立成行。

## 可复用规则
`.env` 值行禁止行内注释（注释独立成行）；遇到「A 路通 B 路不通」的环境型薛定谔故障，先怀疑各路配置解析差异，让故障进程自曝实际 env 是最快的一刀。
