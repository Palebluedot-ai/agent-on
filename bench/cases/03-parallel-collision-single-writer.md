# 案例 03:裸并行撞车 → 单写者+worktree 六步协议

> 层级:L2 教训 | 来源:Euan 实战 2026-07-03 | 入册:2026-07-07 批三

## 症状
想靠多 agent 并行提速,结果两轨互撞:改到同一批文件、抢写同一份状态(progress.yaml),git 与项目状态被搅乱——本项目 fork 撞车实证。反过来单 agent 串行又前后端互相等,快不起来。

## 根因
并行的真问题不在「跑得同时」,在于**共享可变状态无人独占**。裸并行等于放任多个执行者同时写状态、同时改文件;而「接口两侧」本来可以零等待,是因为没先把契约冻死,两轨各自猜对方的形状,一合流就对不上。

## 修法
上六步协议:①契约先行(fixtures+schema 定稿,这步绝不并行)→ ②轨道=目录+git worktree 物理隔离 → ③各轨 fake 对方(对同一份 fixtures 开发,零等待)→ ④契约测试当裁判(双端消费同一 fixture,擅改=两端 CI 同红)→ ⑤单一状态写手(orchestrator 独占 progress.yaml/合并/审查,子代理只干活汇报)→ ⑥合并顺序先契约后实现。派活/收货审查/唯一状态写手这三顶帽子必须在同一个脑子里,拆开就退回撞车。

## 可复用规则
要多 agent 并行前先冻契约(连排序/空值/上限一起冻),每轨一个 git worktree 物理隔离,状态源(progress.yaml)全程只许一个写手改,子代理只干活汇报——单 agent 能干完就别上多 agent。

## 已固化到哪
已固化:六步协议进 `BOOTSTRAP.md` §5「多 agent 并行」;拆成三条铁律 `BOOTSTRAP.md` L3 单一状态写者 + L4 契约先行;`CHARTER.md` 第 2 条把「契约先行并行 / 单一状态写者」列为随骨架生效的方法论不变量;`playbook/multi-contributor-protocol.md` 把同一套不变量从「一人指挥 agent 群」映射到「多人+agent」(单写者位置不动、写者身份从会话换成 maintainer)。
