# 案例 22:编辑器类 MCP 的三层真相——内存 > 磁盘 > 快照

> 层级:L2 教训 | 来源:Euan 实战 2026-07-12(同会话三次复现,commit 3fd26e8/6c70b9e) | 入册:2026-07-12 第四次消化

## 症状
Pencil 里明明改好了,磁盘上的 .pen 文件却是 1.5 小时前的(mtime 实测);同时渲染管线对新节点显示空白,而 batch_get 拿到的数据一直是对的。

## 根因
编辑器类 MCP 工具(Pencil/Figma 一族)改的是**编辑器内存**,磁盘文件要等保存动作才落;快照/渲染又可能滞后于数据模型。三层真相不同步:内存最新 > 磁盘滞后 > 渲染最旧。反直觉处:「文件在仓里」不等于「你看到的版本在仓里」。

## 修法
commit 工件前先查 mtime,过期就请用户在编辑器里保存一次再收;验证走数据模型(batch_get)而非渲染截图——渲染空白 ≠ 稿子坏了。

## 可复用规则
凡带编辑器的 MCP 工具:commit 前查目标文件 mtime,过期先要一次保存;验证以数据模型为准,渲染异常另行排查。

## 已固化到哪
本案例即固化落点(动手前扫坑指针送达);机理对 Figma/任何编辑器型 MCP 通用。
