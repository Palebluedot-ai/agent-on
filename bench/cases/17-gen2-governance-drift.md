# 案例 17:二代标本——防漂移框架自己漂进文档洁癖

> 层级:L2 教训 | 来源:二代仓(communication-governance-playbook) | 入册:2026-07-07 批三

## 症状
一个专门治理"漂移"的框架,自己活成了 14 篇 canonical 文档、MRD 永远卡在 READY_FOR_REVIEW、零实现;锁状态停在 post-boundary-confirmation 再没往前走。更讽刺的是 `01_protocol/` 与 `imported/` 逐字节重复——它违反了自己定的"不许重复真相源"。

## 根因
反直觉处:防漂移的机制(锁、状态机、canonical 文档)本身成了漂移对象。治理文档 ≠ 治理执行;把精力全压在前置治理上,主线一直空转,治理层越修越厚、实现层始终为零。重复真相源不是疏忽,是过度前置治理下没人再看落地、只顾扩建文档的必然产物。

## 修法
不在原仓救,而在下一代换纪律:强模型时代从"锁住 AI"转向"和 AI 对齐"——锁口令体系降级(仅在 phase-gates 保留 fail-closed 思想),改用前置追问 + 契约冻结 + 单写者 + 对抗审查从源头消解漂移。二代的真相源分层、三态转移语义作为正资产提炼进 agent-on,原仓原地归档只留证据。

## 可复用规则
写治理文档前先问"这条规则今天有没有对应的执行或验证":只增 canonical 文档而实现层为零、真相源出现逐字节重复,就是治理空转的信号——治理进 Git 不进聊天,但更要进代码,不是进更多文档。
