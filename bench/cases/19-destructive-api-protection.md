# 案例 19：危险端点先定「什么绝不能删」+ 终态核查脚本验收

> 层级：L2 教训（单项目单次，confidence low）| 来源：Euan 实战 Run #8 | 入册：2026-07-09 首结账消化（intake euan-flutter:destructive-api-protection）

## 症状
设计删除/清号类 API 时，本能是先想「怎么删、删什么」。结果差点让 DEV/系统账号也走进删除路径；而验收时盯着 console 输出「删除成功」，却没法证明该留的对象真的还在、该删的真的没残留。

## 根因
两个错位：①删除逻辑从「删什么」入手，护栏是事后补的，容易漏掉「绝不能删」的对象；②审计/删除类操作的验收信任过程日志，但那些 console 会被输出链（缓冲、截断、并发交错）吞掉，日志说「成功」不等于终态正确。

## 修法
1. **先定护栏端点再定删除逻辑**：先列「什么绝不能删」（DEV / 系统 / 受保护账号一律 403），把这条写成端点级硬拦（Euan `api/src/routes/me.ts` DEV 账号 403 绝对保护），再谈删什么。
2. **验收靠终态核查脚本，不信过程 console**：删完跑一个脚本直接查终态——受保护对象 intact（还在）+ 目标残留 NONE（真没了）。Euan `account-delete.test.ts` + `account-delete.live.test.ts` 双轨核终态。

## 可复用规则
设计删除/清号类 API：先想「什么绝不能删」（护栏端点，系统账号一律 403）再想「删什么」；验收靠终态核查脚本（残留 NONE + 受保护对象 intact），不信过程 console——审计操作的日志会被输出链吞掉。
