# 案例 05:fixture 的隐性契约维度(排序 / 上限 / 空值)

> 层级:L2 教训 | 来源:Euan 实战 2026-07-03（Run #3） | 入册:2026-07-07 批三

## 症状
接口两侧并行,前后端各自对着同一份 JSON fixture 开发,契约测试也在跑,却还是撞了车——两轨对同一份数据的理解不一致,合流时对不上。

## 根因
JSON fixture 明面只锁「形状」(字段名 / 类型),暗面还带着「顺序 / 上限 / 空值语义」这些没写死的维度。Run #3 的撞点=orchestrator 冻结 fixture 时手写的顺序,跟 phase 卡里的排序规则不一致,两轨各信一边,谁都没错却对不齐。反直觉处:形状对齐 ≠ 契约对齐。

## 修法
冻结前跑一个 fixture-lint(排序字段自查),或干脆规定 fixture 一律按响应端(数据的产出方)的排序规则生成,消掉两轨各自解读的缝。

## 可复用规则
冻结 fixture 时把语义一起冻死——排序 / 上限 / 空值口径都要显式写出,别让两轨各自解读。
