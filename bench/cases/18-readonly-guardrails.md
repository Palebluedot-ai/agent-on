# 案例 18：「绝不能写」的场景用三层护栏，不用单点信任

> 层级：L2 教训（单项目单次，confidence low）| 来源：Euan 实战 Run #6 | 入册：2026-07-09 首结账消化（intake euan-flutter:readonly-guardrails）

## 症状
数据迁移轨要读旧库、绝对不能写。单靠「代码里我不会写」这一层信任，一旦某条 SQL 手滑、或某个 ORM 默认行为触发写，就是不可逆的生产事故——而「我小心点」不是护栏。

## 根因
「绝对不能写/删」的操作把安全性全押在单点（写代码的人 / 那一次 review）。单点必然有失守概率，且失守时无兜底、无告警——直到数据已经被改。

## 修法
三层纵深护栏，任一层失守另两层兜底：
1. **运行时层**：只读事务 / 只读数据库连接——物理上发不出写语句。
2. **输入面层**：SQL / 操作白名单——只有列出的读操作能过，其余拒绝。
3. **静态面层**：扫描断言，禁写关键字（INSERT/UPDATE/DELETE/DROP）出现即测试失败。

Euan 迁移轨三层齐上，零误写。

## 可复用规则
任何「绝对不能写/不能删」的操作（数据迁移读侧、审计、只读分析），用三层护栏而非单点信任：运行时（只读事务）+ 输入面（操作白名单）+ 静态面（扫描断言禁写）。拿不准就三层全上——三层的成本远低于一次误写的返工。

## 已固化到哪
本案例卡（bench/cases/18）。与 GStack `careful`/`guard` 类安全 skill 分工：那些管「危险命令执行前拦一下」，本条管「架构层把写能力物理拿掉」——纵深防御，不互斥。
