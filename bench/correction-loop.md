# 修正闭环:发现翻车之后,怎么系统性修掉

> 职责边界:Bench 的案例集(loop-notes / capability-probe)管「发现并记录翻车」;本篇管「发现之后怎么按风险分级地修掉,而不是见虫就改」。不管某个具体 bug 的具体修法。
> 源流:支线 evolving-memory-system/04_protocols/correction-automation-v1.8.md + memory-control-api-v1.7.md,2026-07-07 批二移植(抽思想,不抄它自绑的 Python 管线)

## 一、案例集负责发现,本篇负责修

翻车被记进 `loop-notes.md` 只是第一步——记下来不等于修掉,更不等于以后不再犯。修正闭环补的是后半程:把「发现的问题」变成「按风险分级处理、可追溯、闭得上环」的动作。它和 sop.md 的三桶记账(A 未建功能 / B 疑似缺陷 / C 视觉体验)衔接:三桶负责跑流程时**不见虫就修**、先记账;流程告一段落,这些账就进修正闭环按风险逐条处理。

核心纪律一句话:**改动风险决定谁来拍板——低风险 AI 直接改,中高风险进人工确认队列,不允许 AI 静默改掉碰语义的东西。**

## 二、三态风险分级

每一条待修项先定风险,决定处理路径:

| 风险 | 什么算这一档 | 处理路径 |
|---|---|---|
| **低** | 纯文本 / 文档级:错别字、措辞、注释、格式、失效链接 | AI 直接改,记一笔即可 |
| **中** | 碰配置 / 非核心逻辑 / 有回滚余地的改动 | 进人工确认队列,人拍板后再改 |
| **高** | 碰**代码语义 / 契约 / 数据**:接口行为、fixture 语义、迁移、删测试 | 进人工确认队列,**必须**人确认,禁止 AI 自作主张 |

判档标尺:**改错了能一眼看出来、一步回滚的,是低风险;改错了会悄悄影响别处、或要重跑才发现的,是中高风险。** 拿不准就往高了判——误判成人工只是慢一点,误判成自动改坏了契约是要返工的。

## 三、人工确认队列(最小字段)

中高风险项进队列。队列不需要花哨,一条记这几个字段就够:

| 字段 | 含义 |
|---|---|
| `what` | 要改什么(哪个文件、改成啥) |
| `risk` | 低 / 中 / 高 |
| `decision_reason` | 为什么这么判、为什么这么改 |
| `status` | `待处理` → `已修` / `需人工` |

状态机就两条线:低风险 `待处理 → 已修`(AI 自动);中高风险 `待处理 → 需人工`,人确认后才 `→ 已修`。

**operator 四动作**(谁在管队列,就用这四个动作闭环):
- **处理(resolve)**:确认某项已修,标 `已修` + 一句备注。
- **重排(requeue)**:改了又想重试,打回 `待处理`。
- **拒绝(reject)**:判定这条根本不该修,标掉 + 写原因。
- **列出(list)**:列出所有 `需人工` / `待处理`,给人过目。

## 四、落地形态:一个表格就够

**不要照搬源料那套 Python 自动化管线**——那是它绑死自己仓的东西,agent-on 不做运行时(见 CHARTER 边界)。这一层的正确落地形态是**轻的**:

- 一个 `bench/corrections.md`,里面一张表就是队列,列 = 上面四个字段。
- 低风险 AI 顺手改、表里记一行 `已修`;中高风险 AI 只填一行 `需人工` + `decision_reason`,**不动代码**,等人。
- operator 四动作就是人手改这张表的状态列 + 备注,不需要脚本。
- 复盘节奏挂在切片收口:每次合流(sop.md Phase 7)扫一遍 `corrections.md`,把 `需人工` 的清掉。

**规模化再谈工具**:一张表管不过来了(几十条积压、多人同时改)再考虑抽脚本或做队列文件。在那之前,表格 + 复盘节奏是成本最低、最不容易烂尾的形态——这也符合 CHARTER 的「先固化规则 → 再抽象模板 → 最后自动化」。

一句话带走:**修正闭环的价值不在自动化,在「碰语义的改动 AI 不许自己拍板」这条纪律 + 每条待修项可追溯;一个 markdown 表格就能承载,别一上来就造管线。**
