# 架构师透镜:六维自上而下思考法(含两个现场案例)

> 2026-07-05,用户要求:「作为更高级的架构师,展示你从哪几个维度、如何自上而下思考——落盘让我学习」。
> 本文=方法(六维透镜)+ 两个当场应用(Stripe 支付拓扑 / V1 加固排序)。**先学顺序,再学维度**:顺序本身就是方法——每一维都以上一维的答案为输入。

## 〇、六维透镜(自上而下,顺序固定)

```
① 域与边界 → ② 真相源 → ③ 信任与钱流 → ④ 生命周期完备性 → ⑤ 故障半径 → ⑥ 演进自由度
   (地图)      (谁说了算)   (谁能动什么)     (生老病死齐不齐)    (挂了怎么办)   (明天的门开着吗)
```

**① 域与边界**:先画地图——系统里有哪几个「域」,每域一个职责、一个信任级别、(理想时)一个域名。域边界画对,后面五维的问题会自己变简单;画错,всё 纠缠。
**② 真相源**:每一类事实,指定唯一的「谁说了算」。两处都能写=迟早打架;客户端永远不是真相源。
**③ 信任与钱流**:沿着钱和凭证走一遍——谁能发起、谁验证、密钥住哪、**钱有没有流经我们的服务器**(最好没有)。
**④ 生命周期完备性**:给每个核心实体列「生老病死」表——创建/使用/异常/终结四列,空格=缺口。这是发现「不知道自己不知道」的最高产的一维。
**⑤ 故障半径**:每个外部依赖假设它今晚就挂——用户还能干什么?降级是显式设计还是碰运气?
**⑥ 演进自由度**:今天的每个选择,给明天留了哪扇门、焊死了哪扇门。好架构不是预测未来,是**不焊死未来**。

## 案例 A:Stripe 支付到底怎么跳?官网会不会冲突?(用户 2026-07-05 问)

### ① 域与边界 —— 先看地图,答案已经出来一半

```
euan.pro          = 展示域(纯静态,零登录零密钥,信任级:公开)
api.euan.pro      = 控制域(网关:鉴权/配额/checkout 签发/webhook 接收,信任级:JWT+签名)
checkout.stripe.com = 支付域(Stripe 托管页,信任级:PCI——**不是我们的**)
mail.euan.pro     = 发信域
App               = 交互域(Thin Client,不持任何密钥)
```

**直答**:App 点「升级」→ 网关签发 Checkout Session → App 用系统浏览器打开的是 **checkout.stripe.com(Stripe 自己托管的支付页)——不是官网,也不是我们任何页面**。支付完成 → Stripe 跳到 `api.euan.pro/billing-result.html`(网关上的回执静态页)→ 用户手动回 App 刷新。**官网 euan.pro 从头到尾不在支付链路里**——域已分家,结构上无冲突可言。这正是当初「方案 A:api 子域永久家」买下的红利。

### ②③ 真相源与钱流(为什么这个设计是对的)

- 订阅状态的唯一真相源=**subscriptions 表,由 webhook 写入**——不是支付页回执、不是 App 回报(客户端说「我付了」永远不算数);
- **卡号钱流零经我们**:全程在 Stripe 托管页,我们只拿会话 URL 和签名回调——PCI 合规责任几乎为零。这就是「网关签发凭证,数据面直连」同一哲学在支付上的翻版(和截图直传 Storage 一个道理)。

### ④⑥ 真正需要提前想的两个点(你问「会不会有冲突」——冲突没有,**接口**有两个)

1. **深链回程(v1.1)**:现在支付完回 App 靠「手动切回+下拉刷新」。要做到自动弹回 App,需要 **Universal Links**——而它要求在域名根放一个 `/.well-known/apple-app-site-association` 文件。放在哪个域?建议 euan.pro(主域,一次配置 App 全域深链通用)——**这是官网与 App 唯一的真实接口点:静态官网必须能托管 well-known 文件**。Next.js 的 public/ 目录天然支持(又一个选型论据;拖拽建站工具在这里会卡住)。
2. **T15 IAP 岔路(上架前的硬门)**:TestFlight 阶段外链 Stripe 随便用;**正式上架时 App Store 3.1.1 对 iOS 内数字订阅有约束**,届时三选一:接 IAP / 用外链豁免(按地区政策)/ 仅官网购买。若走「官网购买」,euan.pro 才会第一次长出登录与支付(=SaaS 化立项,brief 已预留)。**今天不用动,但架构上三条路都没焊死**:webhook 汇聚点不变,IAP 只是多一种「订阅事件来源」。

### ⑤ 故障半径(顺手扫)
Stripe 挂 → 升级不了,App 一切照常(计费面与功能面解耦 ✓);webhook 迟到 → 用户付了但 App 没变 paid → 回执页文案已引导「下拉刷新」,webhook 重试机制兜底(幂等已机械证明)✓。

## 案例 B:V1 除了 UI,还要加强什么?(架构师排序版)

方法:拿**④生命周期完备性**和**⑤故障半径**两把尺子扫全系统,凡「表里有空格」的就是加固项——这比按「功能清单」扫多抓一类洞(功能清单只看「有没有」,生命周期表看「全不全」)。

### 生命周期表扫描结果

| 实体 | 生 | 老(使用/维护) | 病(异常) | 死 |
|---|---|---|---|---|
| 账号 | ✓注册+验证 | ✓ | ⚠️ **refresh token 过期路径没人走过**(长期不开 App→?) | ✓删除(S10.1 刚补) |
| 会话 | ✓ | ✓单飞刷新 | ⚠️ 同上;密码改后他端踢出提示无 | ✓销户即失效 |
| 数据(客户/事件/任务) | ✓ | ✓ | ✓错误态齐 | ✓(S10.1 对称端点,**UI 半场未接**) |
| 草稿(记一笔) | ✓ | ✓失败重试 | ⚠️ **离线提交无队列**(断网=只能干等) | ✓ |
| 通知 | ✓排程 | ✓刷新重排 | — | ⚠️ **无开关**(用户关不掉) |
| 新用户体验 | ⚠️ **onboarding 空态=白板**(第一屏没人引导「记一笔」) | — | — | — |

### 故障半径扫描新增

- **App 崩溃=盲区**(R9/T22:无崩溃上报,TestFlight 前必装)
- **回归靠手点**(R10:端到端冒烟缺,当前 636 测试全是模拟环境)

### 加固包提案(S11「生命周期加固」切片,UI 半场合并)

优先级序:①S10.1 的 **UI 半场**(删除/作废/软删入口 + 设置页含通知开关与隐私政策入口 + qa#11/#12)②**会话边角**(refresh 过期平滑回登录不丢草稿——影响每个长期用户的第一印象)③**onboarding 空态**(新用户第一屏即引导)④T22 崩溃监控(可并入)⑤离线写队列(与 🅾 流畅性窗口协调后排)。——全部是 app 侧,**等你两个窗口空档时开工**,后端零改动(S10.1 已备好)。

## 附:架构雷达——非技术 owner 的 unknown-unknowns 解法(Euan 移植)

> 2026-07-09 首结账消化并入(来自 Euan-Flutter `docs/architecture-radar.md`,项目内已运行)。透镜(上文)向下想「这个问题结构对不对」;雷达向前看「什么信号该动、动了要花多少」。**先固化在此,不单开篇目**(冻结令期)。

**要解决的真问题**:AI + 非技术决策者的组合里,最危险的不是「方案选错」,是「不知道自己该做决策」——AI 给一堆建议,owner 看不懂技术权衡、也不知道哪条现在该批。雷达的做法是**用「信号 → 动作 → 量级」三列取代「建议堆」**:owner 不需要理解方案本身,只需认得信号出现、批预算。四件组成:

**1. 每维度一行雷达表**(替代「现状描述」):

| 维度 | 我们在哪 | 主流在哪 | 什么信号该动 | 动的成本(量级) |
|---|---|---|---|---|
| 例:鉴权 | 自签 JWT | 托管 IdP | 团队 >3 人 / 要 SSO | 中(1-2 周) |

owner 读最后两列就够:**信号列 = 什么时候该找我批,成本列 = 批的时候心里有数**。技术权衡压在前两列,不逼 owner 懂。

**2. 地雷清单**(踩了才响的非代码雷):不是「代码里的 bug」,是**架构层踩了才知道的坑**——地域不对齐、平台剥头、第三方精度窗、配额墙。每条一句:雷是什么、踩响的症状、拆雷动作。这一份直接喂 `bench/cases/` 使用时机表(动手前扫)。

**3. 触发式重构预案**:不预测未来,只**把「明天可能要改的门」写下来 + 标触发条件**——「若 X 发生,则按 Y 改,成本 Z」。没触发就不动(避免过度设计),触发了有现成预案(避免临时抓瞎)。对应六维的⑥演进自由度:好架构不焊死未来,雷达是那扇门的登记簿。

**4. 里程碑必扫节律**:每到里程碑(切片收口 / 上线前 / 融资节点),雷达表全表扫一遍——信号列有没有跳、地雷有没有新增、预案有没有该触发的。**挂在节律上,不靠自觉**(同 loop-notes 挂合流 checklist 的思路)。

一句话带走:**透镜是显微镜(把一个问题想透),雷达是望远镜(把该动的信号提前架好);雷达的护城河在「非技术 owner 只认信号、批预算,不必懂方案」——AI 负责把技术权衡翻译成信号与量级。**

## 尾注:这套透镜怎么用在日常

小事不用全走六维——但**凡是「两个系统要打交道」的问题(支付/深链/多端/第三方),先画①域图,再问②真相源**,80% 的「会不会冲突」在前两维就有答案;剩下 20% 藏在④生命周期表的空格里。这也是为什么**雷达(见上文「附」节)管「向前看」,透镜管「向下想」**——一个是望远镜,一个是显微镜。
