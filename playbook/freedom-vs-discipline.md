# 自由度与纪律:为什么一个 Prompt 能出游戏,你的 App 却「感觉受限」

> 2026-07-04,回答 Chao 的两个问题:①怎么让 App 做出来直接就是我想要的?②脚手架会不会把模型能力限制得太死?
> 结论先行:**你感受到的「限制」,大部分不是脚手架的成本,是你产品真实复杂度的成本;但确实有一块自由度被错配了——视觉与探索性工作不该走重流程。解法是二车道。**

## 一、「一个 Prompt 的游戏」是个错觉对照组

拆开看那个游戏 demo 和你的 CRM 差在哪:

| 维度 | 一把梭游戏 | Euan CRM |
|---|---|---|
| 验收标准 | 「跑起来、玩五分钟觉得酷」——**主观、瞬时、无对错** | 金额不能错、数据不能丢、并发不能重复入账——**客观、持久、有对错** |
| 状态 | 内存里,关了就没 | 数据库,错了会留疤,还要跨版本迁移 |
| 集成面 | 0 个外部系统 | Supabase/GoTrue/Grok/Stripe/Resend/APNs/App Store,**7 个**,每个都有自己的脾气(basil 字段搬家、vercel 适配器 POST 挂死——都是本周真事) |
| 用户 | 演示者本人 | 真实用户 + 攻击者(B-token 测试防的就是后者) |
| 决策者 | 模型自己拍板一切 | 你有 21 个 D 决策、4 个用户闸门、若干「以后再聊」 |
| 失败代价 | 重新生成一次 | 掉数据、多扣钱、审核被拒 |

游戏 demo 的「快」,是**把验收标准降到「看起来能跑」换来的**。你的 App 每一分「慢」,几乎都花在游戏不需要的那五行上。这不是模型能力差异,是任务性质差异。**同一个 Fable 让它一把梭写你的 CRM,它也能一小时吐一个「能点的东西」——但金额精度、并发确认、配额原子性这些,会以幻觉的形式埋进去,两周后爆。**

## 二、但你的直觉有一半是对的:自由度确实被错配了

本项目的教训(S0 线框事件):用户期待「好看的 App」,得到的是 token 化线框——因为**探索性/审美性工作被塞进了为「正确性工作」设计的流程**。TDD、契约冻结、验收 checkbox 对「转账不能错」是神器,对「这个页面好不好看」是枷锁:审美没有失败测试可写。

### 二车道模型(本项目即日起生效)

| | 🏎 Explore 车道 | 🚂 Ship 车道 |
|---|---|---|
| 适用 | 视觉稿、交互原型、新功能概念(如 Calendar/Time Page)、技术 spike | 一切碰数据/钱/用户/安全的代码 |
| 流程 | **一把梭**:一个 prompt、Pencil/HTML 原型、可以丢弃、不写测试、不进 progress.yaml | Loop Engineering 全套(phase 卡/TDD/双轨/审查) |
| 产出物 | 「感觉对了」的参照物 | 可上线的代码 |
| 桥 | Explore 的赢家**固化为 Ship 的输入**:视觉稿→design token 更新→UI 卡的验收标准;原型→交互规格 | — |
| 幻觉风险 | 无所谓——原型的幻觉叫创意 | 12 机制封堵(anti-hallucination.md) |

**关键纪律只有一条:两条车道不许串。** Explore 的代码永远不直接 merge 进 Ship(重写,不是搬运);Ship 的流程永远不去管 Explore(不给原型写测试)。

## 三、「怎么让做出来直接就是我想要的」——品味的前置化

复盘本项目走过的弯路,答案是四层「品味注入点」,越早越便宜:

1. **参照物锚点**(最便宜):「客户列表像 Attio、确认卡像 eBay、FAB 像 Things」——一个名词顶一千字描述。**你脑子里的「想要」,大概率已经有个现成产品长那样,说出它的名字。**
2. **Explore 原型投票**(便宜):正差事之前,让模型按 2-3 种方向各出一版可看可点的原型(Pencil/HTML),你选+批注。选择比描述容易十倍——人对「不要什么」的判断远快于「要什么」。
3. **Design token + 视觉稿先行**(中等):选定方向后固化为 token 与 Pencil 稿,此后所有 UI 卡的验收写「按稿」。本项目 token 管道早就位(0 硬编码)但视觉稿缺位——管道好、水没来,这是「UI 差」的根因。
4. **验收标准写「可感知」**(兜底):除了功能 checkbox,加体验 checkbox——「二次进页无转圈」「空态有完成感」。可感知标准让模型知道你在乎什么。

## 四、针对本产品的具体建议

1. **性能切片先行**(已开跑,Run #7):秒开缓存+骨架屏是当前体验短板的最大杠杆。
2. **Calendar / Time Page 走二车道**:这正是 Explore 车道的完美首单——先出 2-3 版概念原型(周视图时间轴?任务+跟进+事件三层日历?今日页的时间流改版?),你选完再立 Ship 卡。**不要直接立功能卡**,日历类功能的形态空间太大,先看再定。
3. **S8 视觉统一改版 = Explore→Ship 的第一次完整走桥**:Pencil 出稿(Explore)→ 你批 → token/稿冻结 → UI 卡按稿重写(Ship,双轨)。
4. 成型判据不变(app-completeness-plan.md §六):3 分钟无文档跑通全流程。

## 五、一句话版

> **游戏一把梭快,是因为它的错误不算错误。你的 App 感觉慢,是因为它的错误真的是错误。把「错误不算错误」的部分(视觉/原型/概念)划进 Explore 车道放飞,把「错误真是错误」的部分(数据/钱/安全)留在 Ship 车道上锁——自由和纪律就都不冤枉。**
