# 元原则:方法论自己怎么保持诚实与进化

> 职责边界:本篇是方法论的方法论——agent-on 自己怎么保持诚实与进化
> 源流:一代/二代/支线 三个前身仓合成,2026-07-07 批二移植

## 引言

> 我们的"酷"不一定非得是这些 skills。我们的"酷"更多的是在我跟你对话的过程中,我的一些想法的沉淀,以及我们如何更好地把这些想法和记忆沉淀下来。

——2026-04-27 深度对话沉淀(`agent-orchestration-playbook/01_thoughts/2026-04-27-deep-dialogue-notes.md`)。这是整个 agent-on 的初心:值钱的不是脚手架的数量,是想法怎么沉淀下来、怎么越沉越好。下面六条,就是为了让这句初心不被后来的复杂度磨掉。

## 一、自己开发自己,必须走自己的流程

**这套方法论在开发它自己时,也必须受同一套门禁约束;如果开发自己还要人工兜底,说明它没解决问题。**

为什么:二代最早写防漂移系统时立过一条硬规矩——它开发自己时也必须做到不漂移,自己的开发流程也走同一条阶段链、文档先落盘再推进、阶段切换可解释可追溯。这不是锦上添花,是核心验证方式。产品口头上承诺"防漂移",开发它自己却靠人手工记忆兜底,承诺和现实就脱节了。agent-on 继承这条:本仓的开发遵守本仓的方法论,决策入 snapshot、完成贴证据、卡片化推进。

出处:二代 `communication-governance-playbook/docs/mrd.md` §13 + `docs/conversation-state-machine-v1.md` §11

## 二、治理契约进 Git,不进聊天记忆

**规则的权威载体是仓里的文件,不是模型的上下文;"这是治理契约,不是提示词技巧。"**

为什么:这句话("This protocol is a governance contract, not a model prompt trick / Store in Git, not only in chat memory")是 2026-05 弱模型时代被逼出来的——当年模型记不住主线,只能把规则钉进 repo,靠文件而非记忆兜底。后来回头看,这恰恰是"模型无关化"的最早表述:规则只要活在 Git 里,换任何一代模型、任何一个工具都还在。偏好可以放模型记忆,操作规则必须放仓库。

出处:支线 `evolving-memory-system/04_protocols/global-inscope-lock-protocol-v1.md` §7

## 三、反思必须产出协议升级,不能只写感想

**每次周期性复盘,至少产出一条对方法论本身的修订——改 playbook/kit 的某个具体文件,并在 changelog/commit 留痕。**

为什么:二代的反射循环里,"反思"这一步不是写读后感,而是硬性要求每周回顾至少产出一条"策略升级"写进协议目录,并落到 `protocol-changelog.md`。方法论不是刻在石头上的教条,是可版本化迭代的活物——复盘只发感慨、不改文件,下次照样踩同一个坑。留痕是为了让"它进化过"这件事可追溯,不是靠记忆声称。

出处:支线 `evolving-memory-system/04_protocols/reflection-loop.md` §4

## 四、上下文边界优先;单 agent 能干完就别上多 agent

**先想清楚上下文怎么切,再谈要不要多角色;编排是成本,不是荣誉。**

为什么:一代编排仓最早的坑就是按"岗位角色"(planner/developer/tester)拆 agent,结果 handoff 里信息一路流失。复盘后立了 ADR-003:先判断上下文边界,再决定 agent 结构——单 agent 能干完且体感不差,就别引入多 agent 复杂度;子任务能干净切开才并行,必须共享状态且相互影响才上 agent team。角色是表现层,上下文边界才是结构层。多开一个 agent 不是本事,是要还的债。

出处:一代 `agent-orchestration-playbook/02_decisions/ADR-003-context-boundary-first.md` + `01_thoughts/2026-04-27-subagent-vs-agent-team-reading-notes.md`

## 五、不许用"感觉更好"替代不变量

**任何变更过门禁,靠可验证的不变量,不靠主观感受——尤其是方法论/流程自身的变更。**

为什么:一代给 AutoPlan 回放机制立过一条红线:"不允许用'感觉更好'替代 invariants。"任何候选策略要上线,先干跑再回放,硬门槛是不变量违规数为 0、高风险审查覆盖率不下降、并发约束不放松,过不了就回滚。灰度也一样,准确率、劣化率、冲突数都是写死的数字阈值。"我觉得这版更顺"不是证据;能被验证的不变量才是。方法论改自己时,这条尤其要守——最容易自我感觉良好的,恰恰是改规则的人。

出处:一代 `agent-orchestration-playbook/05_playbooks/gstack-autoplan-backtest-rollback-v1.md`(红线节)+ `gray-release-checklist-v1.md`

## 六、失败要有预定义降级路径,不许临场即兴

**关键环节在设计时就写好"触发信号 → 降级动作 → 恢复条件",出事时照剧本走,不临场发挥。**

为什么:一代的故障降级 runbook 把审查者不可用、同步故障、锁子系统异常三个场景都提前写死:每个都有明确的触发信号(超时/失败率超阈值)、降级动作(切只读/入队/禁写)、恢复条件(连续健康检查通过、回放成功)。总原则是默认隔离优先于可用性、写失败宁可降级为只读排队也不做无门禁写、每次降级都留审计可恢复。出事那一刻是最不该临时想办法的时候——预案写在平时,现场只执行。

出处:一代 `agent-orchestration-playbook/05_playbooks/failure-drill-degrade-runbook-v1.md`
