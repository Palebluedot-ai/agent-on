# Model Playbook:Fable × Opus 混编经济学

> 2026-07-04,回答 Chao 三问:①两者差在哪 ②给 Opus 补什么脚手架 ③PM 视角怎么少花钱多办事。
> 诚实声明:以下对比不引用跑分(容易编造),只讲**工作负载层面的差异会在哪出现**——依据是本项目七次编排 run 的任务形态分析。

## 一、差异出现在哪(按任务形态,不按玄学)

| 任务形态 | 差异表现 | 本项目对应 |
|---|---|---|
| **长程多约束编排** | 同时持有十几条硬约束(轨道边界/契约冻结/安全红线/用户暂停项)跨小时不漂移——这是层级差的主要体现;约束多到一定程度,弱一档的模型开始「顾此失彼」而非「不会做」 | 主会话当 orchestrator 一整天,零约束违反 |
| **裁决与判断** | 面对矛盾信息(fixture 与卡片打架、平台行为反直觉)先探测再定论,而不是选一个顺手的解释 | Vercel 剥头/CLI 自动确认两案:行为探测→定位→修复 |
| **对抗性审查** | 找茬心态的持续性:审到第 N 项仍保持怀疑,不「前面都对后面大概也对」 | S2 审查抓到被删的并发测试 |
| **边界感知** | 知道什么**不该做**(暂停项/越权操作)并主动停——「负空间」遵守比正空间执行更吃能力 | 加密策略零偷跑;live key 事故即刻刹车 |
| **成熟执行** | 差距最小的地方:写一个明确规格的 CRUD+测试,两者产出难分伯仲 | 七个轨道 agent 的实现质量都很高 |

**一句话:差距不在「会不会写代码」,在「一次能扛多少互相冲突的真相」。** 任务越像「执行明确规格」,差距越小;越像「在模糊中定规格」,差距越大。

## 二、给 Opus 的显性脚手架(能力补齐清单)

原理:上表的每项差距,都可以用**外置结构**补——把「一次扛住」变成「分次扛住」。

1. **约束外置**:phase 卡的 disturbance 栏 + prompt 里铁律编号化(违反=返工)——不指望它记住宪法,每张卡把相关条款抄进来。卡越自包含,模型内存负担越小。
2. **颗粒度降半**:Fable 半天卡 → Opus 给 2-4 小时卡;验收条目 ≤8 条;一卡只让它扛一类约束。
3. **验证外置**:TDD 铁律 + 「完成=贴命令输出」严格执行——把「判断做没做对」从模型脑内移到测试里。
4. **裁决上移**:契约悬点、口径分歧、平台反常行为——卡里写死「拿不准就停下出 Error Signal,不许自行定论」。Opus 执行,裁决权留给上层(Fable 或人)。
5. **审查换人**:执行和审查绝不同一个实例;审查 prompt 给足对抗性(「假定有 bug,找到它」+ 逐条验收核对),并要求**复算证据**而非复述声明。
6. **负空间显性化**:暂停项/禁区不能只在宪法里——每张卡重复相关禁令。「未写明允许即禁止」对 Opus 比「未写明禁止即允许」安全。
7. **探针前置**:接新外部服务的第一张卡固定含「载荷形状对账 + 行为探测」步骤(basil/CLI 事故的制度化预防)——不指望模型「感觉不对」,流程逼它看真实响应。

**注意:这七条就是本项目 SOP 的原样。** 也就是说——**我们的脚手架本来就是按「让任何合格执行者都能跑」设计的,Fable 只是让它跑得更省心,Opus 配齐七条一样能跑。**这正是 orchestration-future.md「保费模型」的实操版。

## 三、混编经济学(PM 的省钱公式)

**原则:贵模型买判断,便宜模型买执行。** 判断的 token 量小、爆炸半径大;执行的 token 量大、有测试兜底。

| 角色 | 模型 | 为什么 | token 占比(经验) |
|---|---|---|---|
| Orchestrator(写卡/冻契约/裁悬点/合流) | **Fable** | 全是判断与多约束;错一处全线返工 | ~15% |
| 轨道 implementer(双轨执行) | **Opus**(配齐七脚手架) | 量大;规格明确;测试+契约兜底 | ~70% |
| 独立审查 | **Fable**(或第二个 Opus 对抗式) | 对抗持续性值钱;但预算紧时 Opus 对抗 prompt 也可用 | ~10% |
| Explore 车道(视觉/概念) | 皆可,甚至更小的模型 | 错误不算错误的域 | ~5% |

**减少反复的四个最大杠杆**(按 ROI 排序,全部已验证):
1. **品味前置**(参照物/画廊投票):消灭「做出来不是我要的」——这是最贵的返工,且与模型无关,是信息问题。
2. **契约语义冻结完整**(排序/空值/上限一起冻):Run #3 撞车 vs Run #4 零返工的全部差别。
3. **验收=机械可判**:每条验收能翻译成测试名,「做完没做完」就不需要讨论。
4. **卡自包含**:换模型/断连/隔天续跑都零摩擦——你的时间不花在「给 AI 恢复上下文」上。

## 四、agent-on 系列的框架沉淀(该搭什么)

把已有六篇+本篇收敛成**四件套资产**,这就是 agent-on 的产品骨架:

1. **📐 Kit(模板层)**:`agent-on/kit/` —— AGENTS.md 骨架、phase 卡模板、轨道 prompt 模板(含七脚手架)、progress.yaml 模板、合流 checklist、目录结构。新项目 clone 即用(sop.md §启动清单的实体化)。
2. **📊 Bench(评测层)**:能力探针任务 ×N + 翻车案例集(basil/微秒窗/适配器/CLI 自动确认…)→ 新模型来了跑一遍,输出「保费档位建议」(哪些环节可降脚手架)。**模型会过时,这个只增值。**
3. **📖 Playbook(方法层)**:现有七篇(sop/anti-hallucination/freedom-vs-discipline/orchestration-future/model-playbook + run-log/loop-notes 数据)——持续由真实 run 喂养。
4. **🧾 Ledger(数据层)**:run-log 的结构化升级(每 run 的模型/时长/冲突/悬点/返工/token 成本)→ 混编经济学的数字依据,也是 agent-on 对外讲故事的证据链。
