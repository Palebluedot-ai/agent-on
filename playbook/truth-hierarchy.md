# 真相源治理:单一不够,还要层级 + 仲裁次序

> 职责边界:本篇管「一个项目里哪份文件算当前真相、冲突时听谁的、引别人的仓时怎么引」,不管具体某份文档怎么写。
> 源流:二代 communication-governance-playbook/docs/source-of-truth-map.md + 支线 evolving-memory-system/04_protocols/ve-governance-adapter-v1.md,2026-07-07 批二移植

## 一、为什么「单一真相源」这句口号不够用

大家都知道要有「单一真相源」(single source of truth)。但真到项目里,单一是个假象——文件天然分三种身份:**现在的规则、迁进来的旧材料、原始证据**。二代吃过一次亏:历史聊天里生成过一批文件,后来被迁移、重命名、归档,新仓又开始重写正式文档。于是每次都卡在同一个问题:该看哪份?imported 里的旧文件还能沿用吗?archive 的 transcript 算不算规范?

所以「单一」要拆成两件事:**层级**(每份文件属于哪一层)+ **仲裁次序**(冲突时哪层压哪层)。缺了仲裁次序,历史材料会悄悄「自动覆盖」当前规则——这是最隐蔽的漂移。铁律一句话:**历史材料不得自动覆盖当前决策。**

## 二、三层怎么在新项目落地

| 层 | 是什么 | 在 agent-on 骨架里对应 | 身份 |
|---|---|---|---|
| **Canonical(当前)** | 当前生效的规则 / 状态 / 契约 | `AGENTS.md`、`docs/`、`docs/state/progress.yaml`、`contracts/` | 唯一权威 |
| **承接(迁入)** | 从旧仓 / 旧上下文搬来的有效材料 | `docs/imported/`(或 `imported/`) | 上游证据,非工作文档 |
| **举证(原始)** | session 归档、日志、transcript、决策快照 | `docs/snapshots/`、`archive/`、`loop-notes.md` | 只负责追溯,不定规则 |

落地规则,三条:

1. **当前只认 Canonical**。产品定义、阶段状态、phase 语义,一律以 Canonical 层为准。承接层或举证层出现不同说法,**不能自动生效**。
2. **收口才算数**。旧材料要沿用,先走一步「收口」:把真正通过的内容誊进某份 Canonical 文档,并说清进的是哪一份。没收口的 imported 文件,永远只是证据,不是工作文档。
3. **判断沿用先问两句**:它进 Canonical 了吗?没进的话该进哪一份?——这两问挡住了「之前是不是聊过、文件到底在哪」的反复回潮。

## 三、仲裁次序(冲突时的硬优先级)

```
Canonical(当前)  >  承接(迁入)  >  举证(原始)
```

- **Canonical 负责当前**:说了算。
- **承接负责承接**:提供素材,不提供裁决。
- **举证负责举证**:证明聊过什么、帮找回上下文,不决定现在怎么推进。

对应到状态机(见 [phase-gates.md](phase-gates.md)),这条落成一个 block 理由码 `BLOCK_SOURCE_OF_TRUTH_CONFLICT`:承接 / 举证材料与当前 Canonical 冲突、又没先完成收口时,拒绝放行。

## 四、跨仓引用:下游只映射不复制

新项目引用另一个仓的规则(最典型:一个新项目引用 agent-on 的方法论)时,最大的坑是**把正文复制过去**——两边各改各的,规则双写漂移,最后没人知道哪份是对的。

正解是**下游只做一张薄薄的映射表,不搬正文**。支线记忆仓的 governance-adapter 已经跑通这套,照搬即可:

1. **pin 版本**:映射文件里写死主规则仓的路径 + 一个 pinned commit(例:`agent-on @ 8f175b4`)。你锚定的是那一刻的规则,不是随时会变的 HEAD。
2. **只映射不复制**:本地文件只写「本仓怎么落地这条规则」(哪些目录可写、冲突归谁),**主规则正文一个字都不抄**。想看原文,顺着 pin 去上游看。
3. **改动走显式升级**:要跟进上游新规则,只做两步——上游先改并提交;下游只更新 `pinned commit` 和必要映射。**永远不在下游改主规则正文。**
4. **跨仓写入要证据**:确需往上游仓回写(比如「agent-on 结账」把沉淀搬回)时,附一张 Promotion Card——source / evidence / confidence / review / rollback / trace 六项齐,缺则拒绝落盘。防的就是随手一改就漂移。

## 五、单读面:给人读的状态面,一个受众只养一份

单写者管「谁写」,这条管「几份」。给人读的状态/待办面(dashboard、真相之页、action list)一个受众只养一份、放显眼处(项目根目录);同受众同类信息面的**第二份自诞生即漂移倒计时**——一律直接删(含指针页),活引用当场改齐,历史存档不改也不救。新开任何人读面之前先答一句「谁在每次变更时喂它」,答不上 = 不开。(Euan 实证:ACTION-LIST 与真相之页并存 6 天即彻底过期,被用户抓包)

## 六、宪法暂停项的局部解禁

Canonical 里「未获指令前禁止」的暂停项允许**局部解禁**,但必须:

1. **D 表划界**:决策记录写清「本次已拍什么 / 仍禁什么」(例:原图 opt-in 已解,真·E2E/KMS 仍禁)。
2. **多面同步**:同批更新 AGENTS 暂停表述、人读仪表盘、TODOS 落地项、威胁模型相关句——禁止只改业务 docs。
3. **禁止聊天默示全解**:会话口头同意 ≠ 整栈能力解禁;执行会话只认文件层划界。

(Euan 2026-07-19 加密暂停项 → D18 局部解禁实证。)

一句话带走:**单一真相源是目标,层级 + 仲裁次序 + pin 映射是让它真的成立的三根钉子。**
