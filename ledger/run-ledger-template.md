# Run Ledger(结构化台账 = Ledger 层)

> run-log 的升级 schema:加模型与成本列,喂养混编经济学。每次合流(merge-checklist 第 7 步)记一行主表 + 一段明细。

## 主表(逐 run 追加)

| Run | 切片 | 日期 | 轨道×模型 | 墙钟 | 串行估 | 冲突 | 悬点(撞/合) | 返工 | tokens(约) | 教训指针 |
|---|---|---|---|---|---|---|---|---|---|---|
| #N | [名] | YYYY-MM-DD | 🔧[模型]×🎨[模型] | Xmin | Ymin | 0 | a/b | n处 | 执行∑ + 编排∑ | loop-notes#锚点 |

<!-- 悬点(撞/合):真撞车数/总悬点数——撞车率是契约冻结质量的直接度量 -->
<!-- tokens:轨道 agent 用量之和 + 主会话本 run 段用量;有 usage 数据就填,没有就估,别空着 -->

## 明细段模板

```
### Run #N — [切片](YYYY-MM-DD)
- 轨道:🔧 [范围](模型 X)× 🎨 [范围](模型 Y)
- 悬点裁决:[逐条:悬点→谁改→为什么]
- 返工:[每处:现象→根因→修法]
- 教训:[一行;展开进 loop-notes]
- 成本备注:[哪轨贵了/便宜了,下次分工调整建议]
```

## 季度汇总问题(Ledger 存在的目的)

1. 撞车率趋势(契约冻结在变好吗?)
2. 返工的根因分布(信息缺口型 vs 能力型——前者改流程,后者调模型档)
3. 单位验收条目的 token 成本,按模型分列(混编比例该动吗?)
4. 探针复跑结论(orchestration-future 的保费档位调整依据)
